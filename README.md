## рҹ”¬ Concept Deep Dives

<details>
<summary><strong>рҹ”№ FIRST & FOLLOW Sets вҖ” Why They Matter</strong></summary>

<br/>

**FIRST(Оұ)** = the set of terminal symbols that can appear as the **first character** of any string derived from `Оұ`.

**FOLLOW(A)** = the set of terminals that can appear **immediately to the right** of non-terminal `A` in some sentential form.

These sets are the mathematical backbone of **LL(1) parsing** and **predictive parsing tables**. Without them, the parser has no way to make a deterministic decision about which production to apply.

```
Grammar:  E вҶ’ T E'
          E' вҶ’ + T E' | Оө
          T вҶ’ F T'
          T' вҶ’ * F T' | Оө
          F вҶ’ ( E ) | id

FIRST(E)  = { (, id }
FIRST(E') = { +, Оө }
FIRST(T)  = { (, id }
FIRST(T') = { *, Оө }
FIRST(F)  = { (, id }

FOLLOW(E)  = { $, ) }
FOLLOW(E') = { $, ) }
FOLLOW(T)  = { +, $, ) }
```

</details>

<details>
<summary><strong>рҹ”№ Shift-Reduce Parsing вҖ” The LR Method</strong></summary>

<br/>

Shift-reduce parsing is a **bottom-up** technique. It reads input left-to-right and builds the parse tree from leaves to root.

**Two fundamental actions:**
- **SHIFT** вҖ” push the next input token onto the stack
- **REDUCE** вҖ” pop a handle off the stack, replace with the grammar's LHS non-terminal

```
Stack           Input           Action
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
$               id + id * id $  SHIFT
$ id            + id * id $     REDUCE: F вҶ’ id
$ F             + id * id $     REDUCE: T вҶ’ F
$ T             + id * id $     REDUCE: E вҶ’ T
$ E             + id * id $     SHIFT
$ E +           id * id $       SHIFT
$ E + id        * id $          REDUCE: F вҶ’ id
$ E + F         * id $          REDUCE: T вҶ’ F
$ E + T         * id $          SHIFT
$ E + T *       id $            SHIFT
$ E + T * id    $               REDUCE: F вҶ’ id
$ E + T * F     $               REDUCE: T вҶ’ T * F
$ E + T         $               REDUCE: E вҶ’ E + T
$ E             $               ACCEPT вң“
```

</details>

<details>
<summary><strong>рҹ”№ Backpatching вҖ” Resolving Forward Jumps</strong></summary>

<br/>

When generating code for `if-else` and `while` loops, the **destination of a jump** isn't known at the time the jump instruction is emitted. **Backpatching** defers filling in the jump target until the destination is known.

```cpp
// Generating code for:  if (x < y) then S1 else S2

E.truelist  = { 100 }     // instruction 100: if x < y goto ___
E.falselist = { 101 }     // instruction 101: goto ___

// After generating S1:
backpatch(E.truelist, nextquad)   // fill in target of true jump

// After generating S2:
backpatch(E.falselist, nextquad)  // fill in target of false jump
```

This is **one of the most elegant algorithms in compiler construction** вҖ” it lets you generate linear single-pass code for structured control flow.

</details>

<details>
<summary><strong>рҹ”№ Flex & Bison вҖ” The Professional Toolchain</strong></summary>

<br/>

**Flex** (Fast Lexical Analyzer) generates a C lexer from regular expression rules:

```lex
%%
[0-9]+      { yylval = atoi(yytext); return NUMBER; }
"if"        { return IF; }
[a-zA-Z]+   { return IDENTIFIER; }
[ \t\n]     ;   /* skip whitespace */
%%
```

**Bison** (GNU Parser Generator) generates an LALR(1) parser from a BNF grammar:

```yacc
%token NUMBER IF IDENTIFIER
%%
stmt : IF '(' expr ')' stmt
     | IDENTIFIER '=' expr ';'
     ;
expr : expr '+' expr   { $$ = $1 + $3; }
     | NUMBER           { $$ = $1; }
     ;
%%
```

Together, they form the standard professional pipeline for **language tool development**.

</details>

```

flex filename.l
# Produces: lex.yy.c

# Step 3: Compile everything together
gcc filename.tab.c lex.yy.c -o parser -lfl

# Step 4: Run the parser
./parser < input.txt
```
